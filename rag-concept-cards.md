# rag-agent · 概念精确卡

> **用途**：消除模糊——每个概念给出可以直接说出口的精确表达，不是「大概就是那种感觉」。
> **格式**：是什么 / 为什么这样而不是别的 / 不是什么
> **使用场景**：学完一关后对照检查、面试前快速过、发现自己说不清某个词时来查。
> **更新规则**：新关卡通关后，把该关涉及的概念对照补全或修正。

---

## RAG（检索增强生成）

**是什么**：把用户问题先送向量库检索相关文档片段，再把片段和问题一起送入 LLM 生成回答的架构——「先找，再答」。

**为什么这样而不是直接问 LLM**：LLM 知识截止、没有私有文档、会幻觉；RAG 把「知识来源」和「语言生成」分离，回答可溯源、可更新知识库而不必重新训练模型。

**不是什么**：不是微调（Fine-tuning）——微调改变模型权重，RAG 不改变模型，只改变输入；也不是全文检索——RAG 用向量语义相似度，不是关键词字符串匹配。

---

## 混合向量检索（Hybrid Search）

**是什么**：同一查询同时走稠密向量（语义相似度）和稀疏向量（关键词权重）两路检索，再用 RRF 按排名融合结果。rag-agent 对应 `node_search_embedding`。

**为什么这样而不是只用稠密向量**：稠密向量对「语义相近但用词不同」强，稀疏向量对「精确关键词」强——单路各有盲区。例如用户问「M2 逾期」，稀疏向量能精确命中「M2」这个术语，稠密向量可能把「逾期客户分级」的泛化语义也召回进来。

**不是什么**：不是两套完全独立的检索系统——BGE-M3 一次 encode 同时产出两种向量，存入 Milvus 同一条记录的两个字段，一次 query 并行跑两路。

---

## BGE-M3

**是什么**：同时输出稠密向量（1024 维浮点）和稀疏向量（高维稀疏权重）的双输出 Embedding 模型，一次 `encode(return_dense=True, return_sparse=True)` 调用获得两种表征。

**为什么这样而不是两个分开的模型**：一次前向传播出两种向量，推理成本只算一次；两种向量共享同一语义空间，对齐程度比两个独立模型更好。

**不是什么**：不是 Reranker——BGE-M3 做的是「文本→向量」（Bi-Encoder，可预计算），Reranker 做的是「问题+文档对→相关度分数」（Cross-Encoder，实时计算），两者用于不同阶段，不可互换。

---

## HyDE（假设文档嵌入）

**是什么**：先让 LLM 根据用户问题生成一段「假设答案」，再用假设答案的向量去检索，而不是用原始问题向量检索。rag-agent 对应 `node_search_embedding_hyde`。

**为什么这样而不是直接用问题向量检索**：「问法」和「知识库里的答法」措辞风格差异大、在向量空间里距离远——用户问「X 是什么意思」，知识库里存的是「X 是一种…可以…」；假设答案的措辞风格和真实答案更接近，向量距离更小，召回更准。

**不是什么**：不是让 LLM 直接回答——假设答案只是检索的跳板，不展示给用户；最终回答仍然基于检索到的真实文档片段生成。

---

## RRF（互惠排名融合，Reciprocal Rank Fusion）

**是什么**：把多路检索结果按排名而非原始分数融合的算法，公式 `score = Σ 1/(k + rank_i)`，k=60 为平滑常数，防止 rank=1 的结果分数过大垄断排名。rag-agent 对应 `node_rrf`。

**为什么按排名而不是按分数加权**：不同召回路的分数量纲不一致——稠密向量余弦相似度在 0-1 之间，稀疏向量 IP 分可能是其他量级，网页相关度分又是另一套；排名是无量纲的，可以跨路对比。

**不是什么**：不是加权平均——加权平均要求分数同量纲；RRF 完全不依赖各路的分数绝对值，同一个 chunk 出现在多路时得分累加，天然实现「多路都觉得好」的提权。

---

## Cross-Encoder Reranker

**是什么**：把「问题+候选文档」拼接后一起送入模型，让问题和文档充分交互后输出精确相关度分数的精排模型。rag-agent 用 BGE-Reranker，对应 `node_rerank`。

**为什么这样而不是用 Bi-Encoder（BGE-M3）直接排序**：Bi-Encoder 问题和文档分别编码再算距离，不能建模两者之间的深层语义交互，精度有上限；Cross-Encoder 让问题「看到」文档内容再打分，精度更高，代价是推理 O(n)、无法预计算。

**不是什么**：不是召回工具——Cross-Encoder 慢（无法对全库预计算），只能对小候选集精排（rag-agent 在 RRF 后取 top-k 再送入）；不能替代 Milvus 的 ANN 召回，两者是串联关系不是替代关系。

## 2 段切片策略

**是什么**：先按 Markdown 标题语义边界切分，再对超长块（>2000 字符）递归拆分、对超短块（<500 字符且同父标题）合并，把所有 chunk 控制在 500-2000 字符区间。rag-agent 对应 `node_document_split`。

**为什么这样而不是直接固定字数切**：固定字数切会在语义中间断裂（一条政策条文从中间截断，前后两块各自语义不完整）；先按标题保语义完整，再按尺寸控 token 成本。500-2000 是「召回精度（偏小）vs LLM 生成质量（偏大）」两股相反力的帕累托区间。

**不是什么**：不是只切一刀——两段是职责分离的两个函数（语义切 + 尺寸精修），语义切不管长短，尺寸精修不管语义。代码块内的 `#` 不触发切分（防止代码行头注释被误识别为标题）。

## item_name（文档级不变量）

**是什么**：从文档正文语义中用 LLM 识别出的业务知识条目名，一份文档切出的所有 chunk 共享同一个 item_name，写入每条 Milvus 记录作为 `WHERE item_name = X` 的过滤条件。rag-agent 对应 `node_item_name_recognition`。

**为什么用 LLM 识别而不是用文件名**：文件名常含版本/日期噪声（`催收办法_v3_final.pdf`）；LLM 从正文抽业务语义级条目名（`《催收管理办法》`），一次识别后广播到全部 chunk，不做 N 次调用。

**不是什么**：不是 chunk 级标签——item_name 是文档级不变量，一份文档所有 chunk 的 item_name 完全相同；它是「在哪个文档里找」的过滤器，不是「这个 chunk 说的是什么」的内容描述。

## LangGraph 图状态机

**是什么**：以 TypedDict 为共享 state、以节点函数为操作单元、以条件边为路由的有向图工作流引擎。每个节点读取并写入 state 字段，图负责按边调度节点执行顺序（包括并行 fan-out）。

**为什么这样而不是顺序函数调用链**：条件路由（PDF vs MD 分支）、并行执行（3 路召回）、错误分支可以在图结构里声明式表达；比嵌套 if/else 函数链更可维护，执行轨迹可观测，节点可独立测试。

**不是什么**：不是微服务或消息队列——LangGraph 是进程内图，不是分布式架构；state 在单次调用生命周期内存活，不持久化。

---

## SSE 流式输出

**是什么**：基于 HTTP 长连接的服务端单向推送协议，格式为 `data: {内容}\n\n`，服务端 `yield` 逐条推送，FastAPI 用 `StreamingResponse(media_type="text/event-stream")` 包装。rag-agent 对应 `node_answer_output` + `app/utils/sse_utils.py`。

**为什么这样而不是 WebSocket**：SSE 是单向推送（服务端 → 客户端），标准 HTTP，无握手，无需维护双向连接状态；问答场景只需服务端推 token，WebSocket 的双向复杂度在这个场景是浪费。

**不是什么**：不是轮询——SSE 服务端主动推，轮询是客户端定时拉，方向相反、延迟特性不同；不是 WebSocket——SSE 客户端无法向服务端推送。

---

## Milvus 双索引

**是什么**：rag-agent 在 Milvus 同一 Collection 内，为稠密向量建 HNSW 图索引（余弦相似度 ANN），为稀疏向量建 SPARSE_INVERTED_INDEX 倒排索引（IP 距离，DAAT_MAXSCORE 算法跳过零维）。对应 `node_import_milvus` 的 schema 建立部分。

**为什么两种向量要用不同索引**：HNSW 适合稠密向量（1024 维全填满，图结构 ANN 快）；倒排索引适合稀疏向量（维度极高但大量为零，只存非零维，若用 HNSW 把所有零维都算进距离，性能和召回都崩）。

**不是什么**：不是两个独立的 Collection——两个索引建在同一 Collection 的不同字段上，一次 query 可以同时触发两路检索，结果在 `node_rrf` 里融合。

---

## MineRU PDF 解析

**是什么**：基于 VLM 的多模态 PDF 解析云服务，输出结构化 Markdown（保留 `#` 标题层级、表格、LaTeX 公式、图片占位符 `![](images/xxx.jpg)`），通过 API 异步调用——提交任务 → 轮询 task_id 状态 → 下载结果。rag-agent 对应 `node_pdf_to_md`。

**为什么这样而不是 pdfplumber**：pdfplumber 按字符坐标硬拼，双栏/表格/公式版式会乱序；MineRU 用视觉模型理解版面语义，正确还原阅读顺序，表格变成 Markdown 表格，公式变成 LaTeX。

**不是什么**：不是本地模型——MineRU 是外部 API 服务，依赖网络和 token；不负责图片内容的语义化——图片占位符只是路径，把图片翻译成可检索文字的工作由 `node_md_img` 里的 VLM 单独完成。
