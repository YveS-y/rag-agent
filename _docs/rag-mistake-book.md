### 🏷 一级标签（答题行为分类）
- `❌不会` — 完全不知道，需要补基础
- `⚠️错位` — 知识点张冠李戴
- `⚠️偏差` — 方向对、细节偏
- `⚠️跑题` — 题目问 A，答到 B 去了

### 📡 二级能力标签（每条必填一个）
- `📝描述力漏洞` — 能识别但描述不清晰（让 AI 写不对）
- `🔍判断力漏洞` — 看不出方案在场景下的坑
- `📐参数直觉漏洞` — 对阈值/数值没有数感
- `⚖️取舍盲区` — 不知道方案间的边界

### #001 · L4 · batch_size三股力方向判断  [⚠️偏差] [🔍判断力漏洞]
📌 原题：GPU显存/吞吐效率/失败重试成本三股力，分别希望 batch_size 偏大还是偏小？
❌ 我的答案：三股力都希望偏小（吞吐效率也归为偏小，理由是"大了会影响IO性能"）
✅ 正确答案：显存、重试成本希望偏小；吞吐效率希望偏大——批量越大，模型调用的固定开销被摊薄到更多条文本上，单条平均耗时更低
💡 关键记忆点：批处理场景里，"调用次数减少、固定开销摊薄"是吞吐效率偏好大批量的核心原因，不要和"显存/重试代价"的方向混在一起
🏷 标签：#node_bge_embedding #batch_size #embedding_utils
📅 首次错日期 / 已回炉次数：2026-06-21 / 0

### #002 · L4 · 参数合理性的验证依据  [⚠️偏差] [⚖️取舍盲区]
📌 原题：怎么验证 batch_size=5 这个值在当前项目里合理？
❌ 我的答案：默认用"三股力权衡"去反推这个值是合理的，没有先确认这个值是不是真的测过
✅ 正确答案：batch_size=5 是直接继承参考代码（老师的实现），从未在本机（Mac M4 统一内存）环境用真实 chunk 数据基准测试过；验证合理性的正确做法是写脚本实测不同 batch_size 下的耗时和内存增量，而不是用理论框架去合理化一个未经验证的继承值
💡 关键记忆点：复用参考代码的超参数时，默认"别人调过了应该没问题"是一个坑；诚实区分"理论上应该怎样"和"有没有真的测过"，敢说"未验证"比编一个解释更有工程判断力
🏷 标签：#node_bge_embedding #batch_size #参数验证方法论
📐 Spec 补洞：L4 Spec 已补上"batch_size 继承未验证，需结合本机硬件实测"这一句
📅 首次错日期 / 已回炉次数：2026-06-21 / 0

### #003 · L8 · RRF为什么用排名而不是原始分数融合  [⚠️偏差] [🔍判断力漏洞]
📌 原题：直接检索路和 HyDE 检索路都做了 norm_score 归一化到 0~1，为什么 RRF 还是不直接拿两路的原始分数加权比较，而要换算成排名（rank）？
❌ 我的答案：先答"奖励两路都召回且得分高的答案"（误以为公式里用了原始分数）；后又答"RRF只是为了缩小范围，真正准确度靠rerank算"（分工没说错，但没答到"为什么不能直接比分数"这个根因）
✅ 正确答案：两路分数数值范围一样（都是0~1），但"含金量"不可比——HyDE 是拿 LLM 生成的"假设性答案"（文档风格的长文本）去匹配真实chunk，比"用户的短问题"去匹配chunk，天然相似度更高，存在系统性虚高。RRF 故意抛弃原始分数，只用每路内部的排名做融合，用同一套"货币"（rank）保证多路可比，避免某一路分数虚高把不那么相关的内容顶到前面
💡 关键记忆点：多路异构召回融合时，"分数范围一样"≠"分数可比"，分数的生成方式（query类型、检索机制）不同就可能有系统性偏差；RRF用rank而非score正是为了规避这个问题，这也是它叫 Reciprocal Rank Fusion 而不是 Reciprocal Score Fusion 的原因
🏷 标签：#node_rrf #RRF #HyDE #多路融合
📅 首次错日期 / 已回炉次数：2026-06-24 / 0

### #004 · L8 · Cross-Encoder为什么不能做召回只能做精排  [⚠️偏差] [🔍判断力漏洞]
📌 原题：BGE-M3（Bi-Encoder）和 BGE-Reranker（Cross-Encoder）计算方式根本不同在哪？为什么 Reranker 不能直接拿来对全库做召回，必须先用 Bi-Encoder 粗筛？
❌ 我的答案：说出了"Bi-Encoder分开算向量、Cross-Encoder拼在一起算"这个表面区别，但坦白"原理还不清楚"——没说出"为什么这个区别会导致一个能扛大规模、一个不能"
✅ 正确答案：Bi-Encoder给chunk生成的向量是在**导入阶段**提前算好存进Milvus的，查询时只需实时算一次问题向量，再和库里已算好的向量做点积/余弦相似度（便宜的数学运算）；Cross-Encoder必须把"问题+chunk"拼在一起才能算分，chunk的表示跟着每次具体问题动态变化，没法提前算好存起来，查询时要对每条候选chunk都重新跑一次完整模型推理——10万条chunk就是10万次推理，资源扛不住。这个性质叫"可预计算性"（precomputability），决定了架构必须是"Bi-Encoder海选→Cross-Encoder精排"，不能反过来
💡 关键记忆点：能不能预计算，是判断一个模型该放在"召回"还是"精排"阶段的核心标准，而不是"哪个更准"
🏷 标签：#node_bge_embedding #reranker_utils #BiEncoder #CrossEncoder #可预计算性
📅 首次错日期 / 已回炉次数：2026-06-24 / 0
