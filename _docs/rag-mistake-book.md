## 📋 错题登记格式

```
### 错题编号 · 关卡 · 问题摘要  [一级/难度] [二级能力]
📌 原题：……
❌ 我的答案：……
✅ 正确答案：……
💡 关键记忆点：……（1-2 句，面试可复用的一句话）
🏷 标签：#节点名 #技术点 ……
📐 Spec 补洞（可选）：如果这条错显示描述力不足，回炉时重写对应 Spec
📅 首次错日期 / 已回炉次数：
```

### 🏷 一级标签（答题行为分类）
- `❌不会` — 完全不知道，需要补基础
- `⚠️错位` — 知识点张冠李戴
- `⚠️偏差` — 方向对、细节偏
- `⚠️跑题` — 题目问 A，答到 B 去了

### 📡 二级能力标签（每条必填一个）
- `📝描述力漏洞` — 能识别但描述不清晰（让 AI 写不对）
- `🔍判断力漏洞` — 看不出方案在场景下的坑
- `📐参数直觉漏洞` — 对阈值/数值没有数感
- `⚖️取舍盲区` — 不知道方案间的边界

## 📚 §1 已登记错题

### #001 · L4 · batch_size三股力方向判断  [⚠️偏差] [🔍判断力漏洞]
📌 原题：GPU显存/吞吐效率/失败重试成本三股力，分别希望 batch_size 偏大还是偏小？
❌ 我的答案：三股力都希望偏小（吞吐效率也归为偏小，理由是"大了会影响IO性能"）
✅ 正确答案：显存、重试成本希望偏小；吞吐效率希望偏大——批量越大，模型调用的固定开销被摊薄到更多条文本上，单条平均耗时更低
💡 关键记忆点：批处理场景里，"调用次数减少、固定开销摊薄"是吞吐效率偏好大批量的核心原因，不要和"显存/重试代价"的方向混在一起
🏷 标签：#node_bge_embedding #batch_size #embedding_utils
📅 首次错日期 / 已回炉次数：2026-06-21 / 0

### #002 · L4 · 参数合理性的验证依据  [⚠️偏差] [⚖️取舍盲区]
📌 原题：怎么验证 batch_size=5 这个值在当前项目里合理？
❌ 我的答案：默认用"三股力权衡"去反推这个值是合理的，没有先确认这个值是不是真的测过
✅ 正确答案：batch_size=5 是直接继承参考代码（老师的实现），从未在本机（Mac M4 统一内存）环境用真实 chunk 数据基准测试过；验证合理性的正确做法是写脚本实测不同 batch_size 下的耗时和内存增量，而不是用理论框架去合理化一个未经验证的继承值
💡 关键记忆点：复用参考代码的超参数时，默认"别人调过了应该没问题"是一个坑；诚实区分"理论上应该怎样"和"有没有真的测过"，敢说"未验证"比编一个解释更有工程判断力
🏷 标签：#node_bge_embedding #batch_size #参数验证方法论
📐 Spec 补洞：L4 Spec 已补上"batch_size 继承未验证，需结合本机硬件实测"这一句
📅 首次错日期 / 已回炉次数：2026-06-21 / 0

## 🔁 §2 回炉计划

**每周五 15 分钟复习**：
1. 随机抽 3 道错题 → 合上答案口述一遍
2. 能完整命中「关键记忆点」→ `已回炉次数 +1`
3. **二级能力维度加测**：抽到 `📝描述力漏洞` → 额外要求 30 字写出对应 Spec；抽到 `🔍判断力漏洞` → 额外要求举 1 个反例场景
4. `已回炉次数 ≥ 3` → 归档到 §3，后续不再抽
