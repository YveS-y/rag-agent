### 错题 #001 · L0 · 离线/在线阶段映射  [⚠️偏差] [📝描述力漏洞]

📌 **原题**：RAG 整体流程分"离线"和"在线"两阶段，对应导入图 / 查询图中的哪一张？判断依据是什么？

❌ **我的答案**：离线建库是导入阶段，**`node_import_milvus` 节点**；在线检索是查询阶段，**`node_web_search_mcp` 节点**。

✅ **正确答案**：
- **导入图整体** = 离线建库阶段（判断依据：所有节点都在"写"——写 MD、写图片描述、写向量）
- **查询图整体** = 在线检索阶段（判断依据：所有节点都在"读"——读向量库、读 MongoDB、向 LLM 要答案）
- `node_import_milvus` 只是导入图里的**最后一步**，不是"离线阶段"本身
- `node_web_search_mcp` 只是查询图里的**一路召回分支**，不是"在线检索"整体

💡 **关键记忆点**：**"阶段 ↔ 整张图，不是阶段 ↔ 单个节点"**——粒度不能错。

🏷 **标签**：#架构俯瞰 #双图模型

### 错题 #002 · L0 · 三级存储分工  [❌不会] [⚖️取舍盲区]

📌 **原题**：Milvus / MongoDB / MinIO 分别存什么？

❌ **我的答案**：client 在图执行前提前创建好，**存 session 信息**？

✅ **正确答案**：
| 存储 | 存什么 | 技术属性 |
|---|---|---|
| **MinIO** | 原始 PDF、MinerU 切出的图片文件 | 对象存储，扛大文件二进制 |
| **MongoDB** | chunk 元数据（`{chunk_id, item_name, parent_title, content}`） | 文档型数据库，嵌套 JSON 天然适配 |
| **Milvus** | 稠密向量 + 稀疏向量 | 向量数据库，HNSW/IVF 索引，扛亿级相似检索 |

💡 **关键记忆点**：**"大文件进 MinIO、结构化文字进 Mongo、向量进 Milvus"**——三者各司一职、零重叠，RAG 系统不存 session 态。

🏷 **标签**：#存储 #Milvus #MongoDB #MinIO

### 错题 #003 · L2 · MinerU 输出格式  [❌不会] [📝描述力漏洞]

📌 **原题**：MinerU 解析一份 PDF 之后，产出的主要数据有哪几类？相比 pdfplumber，在表格 / 公式 / 图文混排三种场景下各强在哪里？

❌ **我的答案**：不知道。

✅ **正确答案**：
- **输出三类产物**：
  1. Markdown 文本文件（`xxx.md`）——保留标题 `#`、段落、表格、LaTeX 公式、图片占位符 `![](images/xxx.jpg)`
  2. 图片资源文件夹（`images/`）——把 PDF 里的图/表截图独立保存为 PNG/JPG
  3. 版式元数据 JSON（`middle.json` / `layout.json`）——记录元素坐标和阅读顺序

- **三场景对比 pdfplumber**：

| 场景 | pdfplumber | MinerU |
|---|---|---|
| 表格 | 按字符坐标硬拼，跨行跨列错位 | 视觉模型识别区域 → Markdown/HTML 表格 |
| 公式 | 只能拿到散乱字符片段 | 识别公式 → 输出 LaTeX，LLM 能理解 |
| 图文/双栏 | 坐标扫描，双栏会左右栏交错 | VLM 理解阅读顺序，不会乱序 |

💡 **关键记忆点**：**"pdfplumber 是几何层面的字符重组；MinerU 是语义层面的版面理解"**——RAG 场景非 VLM 解析器不可。

🏷 **标签**：#mineru #pdf解析 #vlm

### 错题 #004 · L2 · node_pdf_to_md 落盘职责  [⚠️错位] [🔍判断力漏洞]

📌 **原题**：解析产物分别写到 MongoDB 和 MinIO，请对照三级存储分工说说各自写了什么？

❌ **我的答案**：MongoDB 写了 LLM 概括图片内容和图片地址的映射关系；MinIO 存了原始文档和解析后的 MD 文档。

✅ **正确答案**：**`node_pdf_to_md` 根本不直接写 MongoDB / MinIO**！它只做两件事：
1. 落到**本地文件系统**（`local_dir/stem/` 下的图片 + md 文件，中间临时产物）
2. 写进 **LangGraph state**（`md_path` 路径 + `md_content` 内容）

**MongoDB / MinIO 的写入是后续节点做的**（如 `node_md_img`、`node_import_milvus`）。

💡 **关键记忆点**：**"L0 学到的三级存储是整体分工，不是每个节点都全碰"**——具体节点落到哪，翻到函数末尾看 `state['xxx'] = ...` 即可确认。

🏷 **标签**：#node_pdf_to_md #state设计 #知识串位

### 错题 #005 · L2 · 图片处理接力关键步骤  [⚠️跑题] [🔍判断力漏洞]

📌 **原题**：MinerU 产出的 Markdown 里图片以什么形式出现？`node_md_img` 还要多做什么，才能让图片的内容语义进入向量检索？

❌ **我的答案**：向量数据库 → MongoDB 查图片地址 → MinIO 访问图片（这是**查询阶段**的检索链路，不是导入阶段的处理步骤）。

✅ **正确答案**：
- Markdown 里图片形态：`![](images/xxx.jpg)` 的占位符
- `node_md_img` 的动作：**调用视觉 LLM 给每张图片生成文字描述 → 把描述回填到 Markdown 正文里**（替换或紧跟在 `![]()` 后面）
- 这样下一步切片时，图片的语义**变成普通文本的一部分**，一起被 embedding 进向量库

💡 **关键记忆点**：**"embedding 模型只认文本 → 图片必须先被 LLM 翻译成文字 → 回填 MD 后跟正文统一处理"**。导入链路和查询链路要分清：导入只管"塞进向量"，查询才管"取回原图"。

🏷 **标签**：#node_md_img #多模态 #导入vs查询

### 错题 #006 · L3 · 三股力拉扯中召回精度方向搞反  [⚠️偏差] [📐参数直觉漏洞]

📌 **原题**：Embedding token 上限 / 召回精度 / LLM context 成本三股力，各自希望 chunk 偏大还是偏小？为什么？

❌ **我的答案**：
1. Embedding token 上限 → 偏小，越小匹配越精准
2. **召回精度 → 偏大，越大内容越完整** ← 方向搞反
3. LLM context 成本 → 偏小，token 少信息更准确

✅ **正确答案**：

| 拉扯力 | 希望 chunk | 真正理由 |
|---|---|---|
| Embedding token 上限（8192） | **别超就行**（宽松硬线，不是主拉扯力） | chunk<2000 字远远没碰到上限 |
| **召回精度** | **偏小** | chunk 大 → 语义中心分散 → 向量被稀释 → 匹配精度反而下降 |
| LLM 生成质量 | **偏大** | chunk 小 → 信息不完整 → LLM 答不全 |
| LLM context 成本 | 偏小 | top_k=5 × 大 chunk → 快速吃掉 context 窗口 + token 贵 |

**反直觉破除场景**：用户问"逾期 30 天怎么办"，500 字专讲 30 天的 chunk A 比 2000 字涵盖 7/15/30/60/90 五条规定的 chunk B **向量相似度更高**——因为 A 的语义中心就在 30 天这一个点上，B 的语义中心被稀释到 5 个点。

💡 **关键记忆点**：**"chunk 越大 ≠ 召回越好。召回精度希望 chunk 偏小，LLM 生成质量希望 chunk 偏大——这俩是对立方向，500-2000 就是对立拉扯出的帕累托区间。"**

🏷 **标签**：#chunk_size #召回精度 #向量稀释 #参数直觉 #L3

📐 **Spec 补洞**：写 node_document_split 的 Spec 时，必须点明"chunk 目标尺寸 500-2000 是召回精度与 LLM 生成质量的对立拉扯结果"，不能只写"控制 token"。

### 错题 #007 · L3 · Prompt 中反例输入的必要性  [❌不会] [📝描述力漏洞]

📌 **原题**：写一段 Prompt 让 AI 实现「按 # 标题切 Markdown、但代码块里的 # 不切」。要求包含规则 + 反例输入 + 输出契约。

❌ **我的答案**（v1）：只写了规则（行首 1~6 个 # / ``` 包裹跳过），主动表示"不知道什么是反例输入""输出格式没看懂"。

✅ **正确答案**：Prom pt 必备三件套

1. **规则**（抽象约束）：告诉 AI 该怎么做
2. **反例输入**（具体测试用例）：告诉 AI 别怎么错。经典反例：
   - `# coding: utf-8` / `# 作者: xxx`（Python 文件头注释，就算没 ``` 包裹也不是标题）
   - 未闭合代码块（只有开始 ``` 没有结束）
   - `> # xxx`（引用块里的 #）
3. **输出契约**：返回数据的确切形状。经典写法：
   > 以 JSON 数组返回：`[{"title": "...", "content": "...", "parent_title": "..."}]`。不要返回任何解释文字。

💡 **关键记忆点**：**"规则告诉 AI 该怎么做，反例告诉 AI 别怎么错。LLM 看到规则会凭训练分布猜边界 case，反例把高危 case 钉死。Prompt 不写反例，AI 就会瞎猜。"** —— Vibe Coding 铁律。

🏷 **标签**：#prompt工程 #描述力 #代码块防护 #Vibe_Coding #L3

📐 **Spec 补洞**：当你在 Final 关用 Spec 嗂 AI 写项目时，每段 Spec 都要自检三件：规则 / 反例 / 输出契约是否全齐。只写规则的 Spec 是不合格品。

### 错题 #008 · L4 · batch_size 三股力漏掉失败重试成本 + GPU/CPU 混淆  [⚠️偏差] [📐参数直觉漏洞]

📌 **原题**：`node_bge_embedding` 的 `batch_size=5` 是哪 3 股力拉扯出的帕累托点？各自希望 batch 偏大还是偏小？

❌ **我的答案**：
1. 「根据文本大小和 **CPU 性能**平衡」——术语错，应该是 GPU 显存
2. 「100 条 chunk 就要调用 100 次」——✅ 吞吐效率这股识别到了
3. 「50 的话 CPU 消耗更高」——方向对但混了 CPU/GPU
4. **完全没提失败重试成本**——漏了第 3 股力
5. 「为什么是帕累托点是根据经验吧」——把参数直觉当成玄学了

✅ **正确答案**：3 股力拉扯

| 拉扯力 | 希望 batch | 理由 |
|---|---|---|
| **GPU 显存**（硬约束） | **偏小** | batch × 单条 token 数 = 显存占用，太大直接 **OOM 爆炸**，不是"消耗更高"那么温和 |
| **吞吐效率** | **偏大** | 调用次数 = 总量 / batch，batch 越大单位调用开销越低 |
| **失败重试成本** | **偏小** | 1 条挂了不连累其他。batch=50 时 1 条超长/编码异常 → 整批 50 条全赔；batch=5 只赔 5 条 |

**batch=5 = 3 股力拉扯出的帕累托点**，不是"经验"。

💡 **关键记忆点**：**"batch 不是越大越快——越大越脆，一条挂全批赔。GPU 显存是硬约束、吞吐要大、失败重试要小，三股力拉出帕累托区间。"**

🏷 **标签**：#batch_size #GPU显存 #失败重试 #参数直觉 #L4

📐 **Spec 补洞**：Spec v1 也漏写了失败重试成本，v2 才补上。这是第 2 次参数直觉漏洞，周五回炉重点练「3 股力拉扯」模板：任何参数选型都要主动问"还有第 3 股力吗？"

### 错题 #009 · L4.5 · item_name 粒度混淆（chunk 级 vs 文档级）  [⚠️偏差] [📝描述力漏洞]

📌 **原题**：item_name 是 chunk 级元数据还是文档级元数据？一份《催收管理办法》切成 200 个 chunk，它们的 item_name 是 200 个不同还是 1 个广播？

❌ **我的答案**：“item_name 不是每个 chunk 一个么？根据 chunk 内容写的？”——与自己 Q1 答对的「去文件名噪声」逻辑自相矛盾，没认识到 item_name 是文档级不变量。

✅ **正确答案**：

| 维度 | item_name | chunk_id | content / 向量 |
|---|---|---|---|
| 粒度 | **文档级不变量** | chunk 级唯一 | chunk 级 |
| 作用 | L5 `WHERE item_name = X` **文档维度过滤** | chunk 唯一标识 | 内容区分 + 向量检索 |
| 类比 | 身份证号（一家人共同户号） | 个人 ID | 个人特征 |

- **一份《催收管理办法》切成 200 个 chunk → 所有 chunk 的 item_name 均为 `《催收管理办法》`**（单一广播）
- 如果做成 chunk 级（按内容写），L5 查询就**丢掉文档维度的精准隔离能力**：用户问「苹果15 电池如何」，没办法过滤「只找苹果15 说明书」，苹果17 的电池章节会跟着一起返回

💡 **关键记忆点**：**"item_name = 文档身份错点，一份文档所有 chunk 共享同一个 item_name。级别混淆一目了然的验证法：设想 L5 用户问了一句话、LLM 反推 item_name 去过滤 Milvus —— 如果每个 chunk 的 item_name 都不同，WHERE 过滤成了笑话。"**

🏷 **标签**：#item_name #粒度 #文档级不变量 #L4.5 #描述力

📐 **Spec 补洞**：L4.5 Spec v2 必须明确写出「item_name 是文档级不变量」+「一次识别全 chunk 广播」，而不是「给每个 chunk 写个名字」。

### 错题 #010 · L4.5 · 误以为 Milvus 只给 dense_vector 建索引  [⚠️偏差] [🔍判断力漏洞]

📌 **原题**：读 node_import_milvus.py L24–L78 建集合代码时问“为什么只给 dense_vector 创建索引，不给 sparse_vector 创建索引？”

❌ **我的答案**：直接按“只给 dense 建索引”为前提发问、反推设计动机。实际上 L49–L71 明确分两段 add_index，两个字段都建了索引，只是索引类型不同。

✅ **正确答案**：

| 向量类型 | 数据形态 | 索引类型 | 指标/参数 | 为什么选这个 |
|---|---|---|---|---|
| dense_vector | 1024 维浮点全满 | **HNSW** | COSINE · M=32 · efConstruction=300 | 稠密向量的 ANN 主流方案，图结构查得快 |
| sparse_vector | 高维稀疏（大量 0） | **SPARSE_INVERTED_INDEX** | IP · DAAT_MAXSCORE | 倒排只留非零维，整载算分时主动跳过不可能得高分的向量；HNSW 会把 0 也算进距离，性能和召回都崩 |

- 两段 add_index 最后被同一个 index_params 注入到 create_collection，Milvus 会并行维护两套索引。
- 项目因此可做“稠密语义召回 + 稀疏关键词召回”混合检索，最后用 RRF 融合。

💡 **关键记忆点**：**“稠密走 HNSW、稀疏走倒排”——两种向量的数据密度完全不同，必然要用两种不同的索引，只建一个的假设根本站不住脚。**

🏷 **标签**：#milvus #索引 #稠密向量 #稀疏向量 #L4.5 #判断力

📐 **Spec 补洞**：L4.5 Spec v2 必须说清“schema 双字段 ↔ 索引必然双种”的因果，不能只列字段、不讲索引技术选型。
