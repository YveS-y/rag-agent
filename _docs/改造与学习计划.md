## 四、学习成果（已通关，不再是计划，是已完成事项）

> 完整记录见 `_docs/rag-learning-journey.md`（10关+Final Spec集）、`_docs/rag-mistake-book.md`（错题本）、`_learn/进度日志.md`（逐关时间线）。下面只摘录面试可直接复用的结论。

### 4.1 项目完整链路（已验证版本，比早期规划更准确）

**导入链路**：
```
PDF/MD → （图片描述回填）→ 文档切分
       → item_name 识别（LLM取样本调1次，文档级广播）
       → BGE-M3 双模态向量化（dense+sparse）
       → Milvus 入库（9字段schema，先删后插保证幂等）
```

**查询链路**（比早期文档更准确：是 **3 路并行召回**，不是简单 HyDE 单路）：
```
用户问题 → item_name 确认 + 结合历史重写 query
        → 3 级路由（确定/模糊/无匹配商品）
        → 3 路并行召回：
             主路：直接向量检索
             增强路：HyDE 假设性答案检索
             兜底路：MCP 联网搜索
        → RRF 融合（k=60 防止单路结果垄断排名）
        → Cross-Encoder 精排（BGE-Reranker，区别于召回用的 Bi-Encoder）
        → 断崖截断（动态 top-k，非固定数量）
        → SSE 流式输出 + 对话历史存取 MongoDB
```

### 4.2 关键技术概念（错题本沉淀，面试高频）

| 概念 | 一句话记忆点 |
|------|-------------|
| Bi-Encoder vs Cross-Encoder | 召回阶段用 Bi-Encoder（独立编码，快但粗）；精排阶段用 Cross-Encoder（联合编码，慢但准） |
| RRF 的 k=60 | 按排名而非分数融合，k 越大对头部排名的区分度越弱，防止某一路结果直接垄断 |
| 断崖截断 | 不固定 top-k，而是看分数下降的"断崖点"动态截断，避免硬阈值漏掉/多收无关结果 |
| HyDE ≠ 重写 | HyDE 是让 LLM **生成一个假设性的答案文档**再去检索，不是简单改写用户的问题 |
| COSINE / IP / HNSW / 倒排索引 | dense 向量用 HNSW + COSINE；sparse 向量用倒排索引 + IP（内积） |
| item_name 文档级广播 | 同一文档的 item_name 只让 LLM 判断一次（取样本），结果广播给该文档所有 chunk，避免逐 chunk 调用浪费 |

### 4.3 已沉淀的"诚实"类错题（面试加分项：敢说"未验证"比硬编理由更体现工程判断力）

- **batch_size=5 是继承参考代码，从未在本机硬件上实测验证过**——验证方法应该是写脚本测不同 batch_size 下的耗时和内存增量，而非用"三股力权衡"理论去合理化一个没测过的数字。
- `node_item_name_confirm` 文档级 file_title 兜底逻辑是防御性死代码（已识别但暂不处理）。

### 4.4 RAG 面试高频问题 & 答题框架（已用真实链路更新）

**Q1: 你们 RAG 准确率怎么保证？**
> 3 路并行召回（直接检索/HyDE/联网兜底）→ RRF 融合 → Cross-Encoder 精排 → 断崖截断防止噪声。P3 阶段会补充自动化评估的具体通过率数字。

**Q2: 为什么要 3 路召回，不是 2 路或者更多？**
> 直接检索覆盖语义匹配，HyDE 补足"问法和答案表达差异大"的场景，联网搜索做知识库没覆盖到时的兜底——三路分别对应三种召回失败模式，不是越多越好，每多一路都有 RRF 融合的边际成本。

**Q3: Milvus 怎么做混合检索？**
> 同一 Collection 存 dense（HNSW + COSINE）和 sparse（倒排索引 + IP）两套索引，查询时两路并行 ANN，结果送入 RRF 按排名融合。

**Q4: 召回和精排为什么要分两阶段，不直接用精排模型做召回？**
> Cross-Encoder（精排）要把 query 和每个候选文档一起编码，计算量大，无法对全库做；Bi-Encoder（召回）提前把文档编码成向量存起来，查询时只需编码一次 query 做向量检索，可以对全库做，召回候选集后再用精排模型在小范围内做精确排序。

**Q5: 你们项目有什么已知问题但还没修的？**
> item_name 确认的相似度阈值目前写死 0.85，但实际匹配分数常在 0.78-0.83 之间，导致单商品知识库场景下也会触发反问用户的逻辑——这是已识别但优先级排在评估体系之后的待修项，体现了"先有量化指标再决定优化顺序"的工程习惯。
